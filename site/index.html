Edge Security Runtime · Rust

SentinelEdge

Adaptive anomaly detection, energy-aware mitigation, and tamper-evident forensics for resource-constrained devices — packaged as a single Rust binary with no cloud dependency.

This project explores how far a lightweight, self-contained runtime can go toward providing meaningful security guarantees at the network edge. The current prototype implements the core detect–respond–audit loop across six telemetry dimensions. The broader research agenda — zero-knowledge attestation, swarm coordination, post-quantum log integrity — is documented in twenty-five tracks but deliberately deferred until the foundation is stable enough to support it honestly.

Architecture

System Design

The runtime follows a four-stage control loop. Each stage maps to a separate Rust module with a narrow interface. The goal for this milestone was a correct, testable pipeline — not feature breadth.

1

Ingest

Parse typed telemetry samples from CSV. Validate field ranges. Maintain deterministic replay semantics so test scenarios are reproducible.

telemetry.rs
2

Detect

Maintain a rolling EWMA baseline across six signal dimensions (CPU, memory, temperature, bandwidth, auth failures, integrity drift). Compute a weighted anomaly score with human-readable explanations for each contributing signal.

detector.rs
3

Respond

Map the anomaly score to one of four threat levels (nominal, elevated, severe, critical) and a corresponding response action. Degrade gracefully when battery state is constrained.

policy.rs
4

Audit

Append every detection-and-response decision to a chained, hash-linked log. Each entry references the previous, forming a verifiable sequence that can be inspected after the fact.

audit.rs

Pipeline Internals

Telemetry is processed one sample at a time — this mirrors real-time constraints on edge hardware where batch processing is not always available. Below is a closer look at what each stage does and what it deliberately leaves out.

Module Map

Five Rust source files. The prototype is small on purpose — complexity gets added when the research justifies it, not before.

Module Responsibility Tracks
telemetry.rs Input parsing, field validation, typed sample records —
detector.rs EWMA baseline, multi-signal anomaly scoring, explanation generation R01, R05
policy.rs Threat-level mapping, response selection, battery-aware degradation R06, R09
audit.rs Chained audit entries, hash linking, tamper-evident log structure R10, R11
runtime.rs Orchestration, CLI commands, summary rendering —

        Getting Started

        Requirements: a stable Rust toolchain (1.75+). No external services, no database, no cloud account.

        Build

        cargo build

        Run the demo trace

        cargo run -- demo

        Runs a short built-in telemetry sequence. Audit output goes to var/demo.audit.log.

        Analyze a scenario

        cargo run -- analyze examples/credential_storm.csv

        Full ingest → detect → respond → audit pipeline on the included credential-storm trace.

        Run tests

        cargo test

        Formatted report

        cargo run -- report examples/credential_storm.csv

        Implementation snapshot

        cargo run -- status

        Expected CSV Format

        The telemetry parser expects this header row:

        timestamp_ms,cpu_load_pct,memory_load_pct,temperature_c,network_kbps,auth_failures,battery_pct,integrity_drift
